# ResumeIQ — AI Resume Analyzer & ATS Score Predictor

An end-to-end Deep Learning + rule-based system that analyzes resumes using a
**Bidirectional GRU** (TensorFlow/Keras) for classification, combined with a
fully separate, explainable **ATS Engine** for skill matching, quality
checks, and improvement recommendations.

> **Design constraint:** This project intentionally uses **only GRUs** to
> demonstrate recurrent Deep Learning architectures. No BERT, RoBERTa,
> Sentence-Transformers, Hugging Face models, or LLM APIs are used anywhere
> in the pipeline.

---

## 🧠 What the GRU does (and doesn't do)

The Bidirectional GRU is a **pure text classifier** with 3 output heads:

| Head | Predicts |
|---|---|
| `category` | Resume job category — 24 real classes from the dataset (Information Technology, HR, Sales, Finance, Engineering, Teacher, Chef, Aviation, Healthcare, etc. — see full list below) |
| `experience_level` | Fresher / Junior / Mid-Level / Senior (see **heuristic label** note below — this is not ground truth) |
| `job_domain` | Mirrors `category` (this dataset has no independent domain signal — kept as a 3rd head for architectural symmetry with the original design) |

**The GRU does NOT generate ATS suggestions.** All scoring, skill matching,
formatting checks, and recommendations are handled by a **separate,
rule-based ATS Engine** (`ats_engine/`) so every suggestion is traceable to
an explicit rule — not a black-box prediction.

## 🏗️ Architecture

```
Upload Resume (PDF/DOCX)
        │
        ▼
Text Extraction (PyMuPDF / pdfplumber / python-docx)
        │
        ▼
Job Description Input (paste / sample / upload)
        │
        ▼
NLP Preprocessing (lowercase → remove symbols → remove stopwords → lemmatize)
        │
        ▼
Tokenization + Sequence Padding (Keras Tokenizer)
        │
        ▼
Embedding Layer (trainable, 128-dim)
        │
        ▼
Bidirectional GRU (128 units) → Bidirectional GRU (64 units)
        │
   ┌────┼────────────────┬─────────────────┐
   ▼    ▼                ▼                 ▼
Category            Experience Level   Job Domain
(Dense+Softmax)      (Dense+Softmax)   (Dense+Softmax)
   │                     │                 │
   └─────────────────────┴─────────────────┘
                    ▼
          Structured Resume Profile
                    │
      ┌─────────────┴──────────────┐
      ▼                             ▼
Skill Extraction Engine     Resume Rule Engine
(dictionary + regex)        (sections, weak verbs,
                              passive voice, metrics,
                              contact info, length)
      │                             │
      └─────────────┬───────────────┘
                     ▼
             ATS Matching Engine
     (resume skills vs. JD skills → match %)
                     │
                     ▼
             ATS Score Calculator
     (weighted: skill match + quality score)
                     │
                     ▼
          Recommendation Engine
  (missing skills, missing sections, weak
   summary, project ideas, formatting fixes)
                     │
                     ▼
          Streamlit Dashboard
  (scores, gauges, skill match chart,
   strengths, suggestions, PDF/CSV export)
```

## 🚀 Usage

### 1. Prepare the training dataset (already done — included in this zip)

The raw Kaggle CSV has already been converted for you. To regenerate it, or
if you swap in your own raw file at `datasets/raw_kaggle_resume_dataset.csv`:

```bash
python datasets/prepare_kaggle_dataset.py
```

### 2. Train the GRU model

```bash
python -m models.train --epochs 12 --batch_size 32
```

This produces `saved_model.keras`, `tokenizer.pkl`, `label_encoders.pkl`,
`training_history.json`, and `evaluation_report.json` inside `models/`
(already included, pre-trained, in this zip — training takes ~15 min on
CPU for this dataset size).

### 3. Run inference from the command line (optional, for testing)

```bash
python -m models.predict
```

### 4. Launch the Streamlit dashboard

```bash
streamlit run streamlit_app/app.py
```

Then in the browser:
1. Upload a resume (PDF or DOCX) in the sidebar.
2. Paste a job description, pick a sample, or upload one.
3. Click **Analyze Resume**.
4. Review the GRU predictions, ATS score, skill match chart, strengths, and
   suggestions — then download the PDF/CSV report.

*(Add your own dashboard screenshots here after running it locally, e.g.
`docs/screenshot_dashboard.png`.)*

## 📊 Model Evaluation

Evaluation metrics (accuracy, precision, recall, F1, confusion matrix) are
computed automatically at the end of `models/train.py` and saved to
`models/evaluation_report.json`. Actual results from training on the real
Kaggle dataset (2,483 resumes, 20% held out as a test set, 24 categories):

| Output Head | Accuracy | Precision (weighted) | Recall (weighted) | F1 (weighted) |
|---|---|---|---|---|
| `category` | 0.694 | 0.693 | 0.694 | 0.680 |
| `experience_level` | 0.847 | 0.718 | 0.847 | 0.777 |
| `job_domain` | 0.708 | 0.714 | 0.708 | 0.695 |

Context: `category` at 69% across **24 classes** (random baseline ≈4%) with
only ~2,483 samples and no pretrained embeddings is a reasonable result for
a from-scratch GRU — training accuracy reaches 95%+, so the gap versus
validation reflects the dataset being small relative to the class count
(~100 samples/class) rather than a modeling bug. `experience_level`'s higher
accuracy is inflated by the Senior-heavy label skew described above (its
lower precision reflects the model struggling on minority classes).

Full per-class confusion matrices and classification reports are in
`models/evaluation_report.json`.

## ⚠️ Important notes on the training dataset

This project now trains on the **real Kaggle "Resume Dataset"**
(`datasets/raw_kaggle_resume_dataset.csv` → converted by
`datasets/prepare_kaggle_dataset.py` → `datasets/resume_dataset.csv`):
**2,483 real resumes across 24 categories** (Information Technology, HR,
Sales, Finance, Business Development, Advocate, Chef, Engineering,
Accountant, Fitness, Aviation, Healthcare, Consultant, Banking,
Construction, Public Relations, Designer, Arts, Teacher, Apparel, Digital
Media, Agriculture, Automobile, BPO).

The raw file only has columns `ID, Resume_str, Resume_html, Category` — it
provides real, human-labeled ground truth for `category` only. Two
adjustments were needed to fit the project's 3-head architecture:

1. **Column mapping**: `Resume_str` → `resume_text`, `Category` → `category`
   (cleaned to title case, e.g. `INFORMATION-TECHNOLOGY` → `Information
   Technology`).
2. **`experience_level` is a derived/heuristic label**, NOT ground truth.
   `prepare_kaggle_dataset.py` infers it by scanning resume text for
   explicit "X years of experience" mentions and seniority keywords
   (Senior/Lead/Manager vs. Intern/Entry-level), falling back to resume
   length if neither is present. **Known limitation**: because many resumes
   in this dataset open with boilerplate like "15+ years of experience...",
   the derived labels skew heavily toward "Senior" (~83% of the dataset) —
   evaluation accuracy on this head should be read as *"how well the GRU
   reproduced the heuristic,"* not as true seniority-prediction accuracy. If
   you have or can source real seniority labels, replace this column for a
   much more meaningful signal.
3. **`job_domain` mirrors `category`** — this dataset has no independent
   domain field, so the third head is kept only for architectural symmetry
   with the original 3-head design. In practice `category` and `job_domain`
   predictions will always agree for this dataset.

**To regenerate `resume_dataset.csv` from the raw file** (e.g. after editing
the heuristic):
```bash
python datasets/prepare_kaggle_dataset.py
```

**To use a different dataset entirely**, any CSV with these 4 columns works
with `train.py` unchanged:
```
resume_text, category, experience_level, job_domain
```

**Also note**: `datasets/skills_database.json` was originally tech-only. It
now includes a few non-tech categories (HR, Sales, Finance) added to support
ATS skill-matching for those resume types, but it does **not** yet cover all
24 categories (e.g. Chef, Aviation, Teacher have no dedicated skill lists).
ATS skill-match scores for those categories will be sparse/low until the
dictionary is expanded — this is independent of the GRU model itself.

## 🧩 Key Design Decisions

- **GRU over LSTM**: chosen for faster training and fewer parameters, which
  suits a smaller resume dataset (see `models/gru_model.py` for how easily
  this can be swapped to `LSTM` if you scale up the dataset later).
- **Separate ATS Engine**: keeping scoring/recommendations rule-based (not
  learned) means every suggestion shown to the user is explainable and
  auditable — a resume never gets a mysterious deduction with no reason
  attached.
- **No pretrained/Transformer embeddings**: `data_preprocessing/embedding.py`
  supports optional GloVe loading, but the default and demonstrated path is
  a trainable embedding layer learned end-to-end with the GRU, keeping the
  whole project GPU-free and self-contained.
